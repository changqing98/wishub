# wishub（智慧中枢）MVP 线框图说明

本文档用于将页面原型说明进一步压缩为适合产品评审与设计起稿的线框图说明稿。重点不是视觉风格，而是：**页面结构、模块优先级、信息层次与交互路径**。

---

## 1. 文档目标

本说明主要用于：

1. 帮助产品快速 Review 页面结构是否合理
2. 帮助设计进入中低保真原型绘制
3. 帮助研发理解页面模块之间的关系
4. 在正式视觉设计前，先锁定信息架构与交互骨架

---

## 2. 页面范围

wishub（智慧中枢）mvp 版本建议至少绘制以下 3 个核心页面：

1. 外部客户问答页
2. 证据在线预览页 / 预览弹层
3. 知识库管理页

可选补充页面：
- 文件上传结果页 / 上传结果区域
- 空状态页
- 异常状态页

---

## 3. 页面一：外部客户问答页线框说明

## 3.1 页面目标
- 让客户一眼知道这是“基于知识库的问答”
- 让客户快速提问
- 让客户看懂系统回复结果类型
- 让客户快速查看证据
- 让客户快速回看最近提问记录

## 3.2 信息优先级
建议从高到低排序为：
1. 问题输入
2. 系统回答结果
3. 证据卡片
4. 最近提问记录
5. 使用说明

---

## 3.3 桌面端线框草图

```text
┌────────────────────────────────────────────────────────────────────┐
│ Logo / 产品名：wishub（智慧中枢）                                 │
│ 说明：基于知识库回答问题；问题不明确时会提示补充；无依据时会明确说明 │
├───────────────────┬────────────────────────────────────────────────┤
│ 最近提问记录       │ 主问答区                                       │
│                   │                                                │
│ - 最近提问         │ 用户问题气泡                                   │
│ - 问题摘要 1       │ 系统状态条（检索中 / 需补充 / 已回答 / 不知道） │
│ - 问题摘要 2       │ 系统回答卡                                     │
│ - 问题摘要 3       │ - 答案                                         │
│ - 重新提问         │ - 说明                                         │
│                   │ - 证据卡片列表                                 │
│                   │   - 文档名                                      │
│                   │   - 路径                                        │
│                   │   - 片段                                        │
│                   │   - 查看引用位置                                │
├───────────────────┴────────────────────────────────────────────────┤
│ 输入框：请输入你想了解的问题……                        [提问按钮]     │
└────────────────────────────────────────────────────────────────────┘
```

---

## 3.4 模块说明

### 模块 A：产品说明区
建议放在页面顶部，内容简短，不超过 2~3 行。

必须表达清楚 3 个边界：
- 基于知识库回答
- 有证据才回答
- 无依据会明确说明

### 模块 B：最近提问记录
建议放在左侧，作为辅助区，而不是主操作区。

展示重点：
- 当前用户最近问题摘要
- 点击可复用
- 长文本省略

不要做成：
- 复杂搜索器
- 多级筛选器
- 运营后台样式

### 模块 C：问答结果区
是页面最核心区域。

建议结构顺序：
1. 用户问题
2. 状态条
3. 系统答案卡
4. 证据列表

### 模块 D：输入区
应固定在页面底部或主区域下方。

必须保证：
- 输入框足够明显
- 提问按钮位置稳定
- 移动端时也不影响输入体验

---

## 3.5 状态表现建议

### 情况 A：有依据回答
表现：
- 状态条：已根据知识库找到相关答案
- 显示答案卡
- 显示证据卡片

### 情况 B：无依据拒答
表现：
- 状态条：当前知识库中没有足够信息
- 显示拒答文案
- 不显示伪证据卡片

### 情况 C：问题不清晰
表现：
- 状态条：需要补充问题
- 显示澄清提示卡
- 不显示正式答案卡

### 情况 D：系统异常
表现：
- 状态条：当前暂时无法完成回答
- 显示重试型异常提示
- 与“不知道”状态强区分

---

## 3.6 移动端适配建议

在移动端建议改为上下结构：

```text
┌──────────────────────────────┐
│ 产品说明                     │
├──────────────────────────────┤
│ 最近提问（横滑或折叠）       │
├──────────────────────────────┤
│ 问答结果区                   │
│ - 用户问题                   │
│ - 系统结果                   │
│ - 证据卡片                   │
├──────────────────────────────┤
│ 输入框 + 提问按钮            │
└──────────────────────────────┘
```

原则：
- 证据卡片保持可点
- 最近提问可折叠
- 输入区不能被遮挡

---

## 4. 页面二：证据在线预览页 / 预览弹层线框说明

## 4.1 页面目标
- 让用户快速验证答案出处
- 强调“仅查看引用内容”
- 防止整篇文档暴露

## 4.2 建议形态
优先建议使用：
- 桌面端：右侧抽屉或中型弹层
- 移动端：全屏层

这样更符合“查看证据”而不是“跳转离开主流程”。

---

## 4.3 线框草图

```text
┌────────────────────────────────────────────────────┐
│ 文档名：开发者接入指南.pdf        页码：12     [关闭] │
│ 路径：/docs/openapi/开发者接入指南.pdf              │
├────────────────────────────────────────────────────┤
│ 提示：当前仅展示与该回答相关的引用内容               │
├────────────────────────────────────────────────────┤
│ 片段 1（高亮）                                      │
│ 回调地址必须为 HTTPS，并且可以被平台访问。          │
│                                                    │
│ 片段 2（上下文）                                    │
│ 完成配置后，需要在控制台进行回调验证。              │
├────────────────────────────────────────────────────┤
│ [上一处命中]                       [下一处命中]       │
└────────────────────────────────────────────────────┘
```

---

## 4.4 模块说明

### 模块 A：头部信息区
必须展示：
- 文档名
- 路径
- 页码或段落定位信息
- 关闭操作

### 模块 B：预览提示区
必须明确提示：
- 仅展示引用内容
- 不提供完整原文浏览

### 模块 C：片段区
展示优先级：
1. 命中片段
2. 上下文片段
3. 高亮效果

### 模块 D：命中切换区
在同一个回答有多个命中点时使用。

注意：
- 切换逻辑要简单
- 不要演化成复杂文档浏览器

---

## 4.5 评审重点

Review 时重点看：
- 用户一进来是否能立刻看到命中内容
- 页码信息是否足够明确
- 是否存在可继续浏览整篇文档的路径
- “证据核验页”是否被误做成“文档阅读页”

---

## 5. 页面三：知识库管理页线框说明

## 5.1 页面目标
- 让内部运营能方便导入和维护知识资料
- 清晰展示目录结构
- 清晰展示文件状态
- 支持删除与替换

## 5.2 信息优先级
建议从高到低排序为：
1. 上传入口
2. 目录树
3. 文件列表与状态
4. 文件操作
5. 结果提示

---

## 5.3 桌面端线框草图

```text
┌────────────────────────────────────────────────────────────────────┐
│ 页面标题：知识库管理                                [上传文件] [上传文件夹] │
├────────────────────────┬───────────────────────────────────────────┤
│ 目录树                  │ 文件列表 / 文件详情                       │
│                        │                                           │
│ root                   │ 文件名 | 类型 | 状态 | 更新时间 | 操作     │
│ ├─ docs                │ ----------------------------------------  │
│ │  ├─ openapi          │ 开发者接入指南.pdf | PDF | READY | ...   │
│ │  ├─ faq              │ 常见问题.txt       | TXT | READY | ...   │
│ ├─ product             │ 退款规则.md        | MD  | FAILED| ...   │
│                        │                                           │
│                        │ 操作：删除 / 替换                         │
└────────────────────────┴───────────────────────────────────────────┘
```

---

## 5.4 模块说明

### 模块 A：顶部操作区
必须包括：
- 上传文件
- 上传文件夹
- 可选刷新状态

### 模块 B：目录树
应支持：
- 展开 / 折叠
- 当前选中高亮
- 层级结构清晰

### 模块 C：文件列表区
建议展示字段：
- 文件名
- 文件类型
- 状态
- 更新时间
- 操作项

### 模块 D：操作区
至少支持：
- 删除
- 替换

不建议首版加入：
- 批量移动
- 批量编辑
- 标签系统

---

## 5.5 上传结果展示建议

### 单文件上传
- 顶部或列表局部显示上传状态
- 文件进入当前目录

### 文件夹上传
- 优先展示整体任务状态
- 同时在目录树中逐步显现结果
- 若部分失败，要能看出哪些失败

### 失败文件
必须显眼区分，不要与正常文件状态混在一起。

---

## 5.6 评审重点

Review 时重点看：
- 上传按钮是否足够明显
- 目录树是否足够清晰
- 文件状态是否一眼可分辨
- 删除/替换操作是否低学习成本
- 部分成功场景是否会误导用户

---

## 6. 核心组件线框说明

## 6.1 证据卡片组件
建议结构：

```text
┌────────────────────────────────────┐
│ 文档名：开发者接入指南.pdf         │
│ 路径：/docs/openapi/...            │
│ 片段：回调地址必须为 HTTPS ...      │
│                    [查看引用位置]  │
└────────────────────────────────────┘
```

### 评审重点
- 文档名是否醒目
- 路径是否有助于理解来源
- 片段长度是否适中
- CTA 是否足够明确

---

## 6.2 澄清提示卡组件
建议结构：

```text
┌────────────────────────────────────┐
│ 为了更准确地回答你，请补充一下：   │
│ - 具体产品 / 功能                  │
│ - 页面名称 / 操作步骤              │
│ - 报错信息（如有）                 │
└────────────────────────────────────┘
```

### 评审重点
- 是否足够友好
- 是否真正帮助用户继续提问
- 是否与“拒答卡”区分明显

---

## 6.3 拒答卡组件
建议结构：

```text
┌────────────────────────────────────┐
│ 当前知识库中没有足够信息回答这个问题 │
│ 你也可以换一种更具体的问法试试。     │
└────────────────────────────────────┘
```

### 评审重点
- 是否足够克制
- 是否避免让用户误以为系统出错

---

## 7. 页面流转关系

```text
客户问答页
  ├─ 提问成功 → 回答成功卡 + 证据卡片
  │                   └─ 点击证据 → 在线预览层
  ├─ 提问成功 → 拒答卡
  └─ 提问成功 → 澄清提示卡

知识库管理页
  ├─ 上传文件 → 文件状态变化
  ├─ 上传文件夹 → 目录树更新
  ├─ 删除文件 → 文件移除
  └─ 替换文件 → 文件状态刷新
```

---

## 8. 产品 Review 建议顺序

建议你在 Review 线框时，按下面顺序看：

1. 信息架构是否清楚
2. 主任务路径是否顺畅
3. 状态区分是否足够明显
4. 证据链路是否可信
5. 原文边界是否被严格控制
6. 运营维护路径是否简单可用

---

## 9. 设计交付建议

如果这份线框说明通过 Review，设计侧建议按以下顺序产出：

1. 客户问答页低保真
2. 在线预览抽屉 / 弹层低保真
3. 知识库管理页低保真
4. 核心组件（证据卡、澄清卡、拒答卡）
5. 再进入中保真 / 高保真设计

---

## 10. 本稿 Review 重点结论

这份线框说明最关键要确认的是：

- 问答页是否真正突出“答案 + 证据”双核心结构
- 预览是否足够方便核验，但又不会变成全文阅读器
- 管理页是否足够轻量，适合 MVP 快速交付
- 所有关键状态是否都能被页面结构承接
