# wishub（智慧中枢）MVP 正式 PRD

> 版本：MVP  
> 文档用途：作为产品、设计、研发、测试和验收的统一依据。  
> 核心原则：知识库是唯一事实来源；有依据才回答，无依据明确拒答。

## 1. 背景与目标

### 1.1 背景
企业已有的产品说明、使用指南和技术资料分散在文档中，用户需要通过自然语言快速获得准确答案。MVP 先验证“导入 Markdown 知识—查看目录规模—基于知识库问答”的最小闭环。

### 1.2 业务目标
- 建立可用于问答的 Markdown 知识库。
- 让用户能直观看到知识库当前的文档目录数。
- 提供严格基于知识库的知识问答，降低错误或幻觉回答。

### 1.3 用户价值
- 用户通过自然语言获取知识库内的答案。
- 用户在问题无依据时得到明确、可信的拒答或澄清提示。
- 知识维护者能确认文档是否成功进入知识库，并了解目录规模。

### 1.4 成功指标
- Markdown 文档可成功上传并进入可问答状态。
- 目录数展示与系统实际目录计数一致。
- 有知识库依据的问题返回相关答案；无依据问题不输出知识库外结论。
- 上传、处理和问答的关键异常均有明确反馈。

## 2. 用户与场景

### 2.1 目标用户
- 知识维护者：上传 Markdown 文档并确认导入结果。
- 知识使用者：基于知识库查询产品、流程或技术信息。

### 2.2 核心场景
1. 知识维护者上传一份 Markdown 文档，系统校验格式并处理文档。
2. 知识使用者查看知识库文档目录数，判断当前知识覆盖规模。
3. 知识使用者提出知识库内问题，系统检索并生成有依据的回答。
4. 知识使用者提出知识库外或证据不足的问题，系统拒答并说明原因。
5. 知识使用者提出语义不完整的问题，系统要求补充必要信息，不进行猜测。

### 2.3 用户痛点
- 资料分散，定位信息成本高。
- 不清楚知识库是否已导入或当前规模。
- 通用模型可能凭空补充，难以判断答案可信度。

## 3. 范围定义

### 3.1 本期范围
- Markdown 单文件上传。
- Markdown 文档解析、入库及处理状态反馈。
- 知识库文档目录树/目录列表的查看。
- 知识库文档目录数展示。
- 基于当前知识库的自然语言问答。
- 有依据回答、依据不足拒答、问题不清晰时澄清。
- 问答结果展示引用文档名称或引用片段（以实际可用的证据定位能力为准）。

### 3.2 明确非本期范围
- PDF、TXT、Word 或其他非 Markdown 文件。
- 文件夹上传、批量目录导入、压缩包导入。
- 文件删除、替换、编辑、重命名、移动、版本管理。
- 原文预览、全文阅读、页码定位或文档下载。
- 用户权限、角色权限、登录体系、多租户和组织管理。
- 复杂历史会话、会话管理、跨会话记忆和多用户隔离。
- 知识库外自由问答、联网搜索、常识补答。
- 人工客服转接、消息通知和高级运营分析报表。

## 4. 产品原则与通用规则

### 4.1 事实边界
- 知识库是 MVP 唯一允许使用的事实来源。
- 模型不得使用知识库之外的训练知识、联网内容或臆测补全作为答案事实。
- 回答必须与用户问题相关，不能仅因命中关键词就输出无关片段。

### 4.2 证据规则
- 有效回答必须建立在检索到的知识库内容之上。
- 能展示证据时，至少展示引用文档名称；若系统支持片段返回，则同时展示相关引用片段。
- 证据不足以支持完整结论时，只回答证据能够支持的部分，或直接拒答。
- 不得伪造文档名称、引用片段、链接或证据位置。

### 4.3 拒答与澄清规则
- 知识库没有相关内容时，明确告知“当前知识库暂无相关依据”，不得编造答案。
- 问题存在多个可能含义、缺少对象或关键条件时，先提示用户补充，不猜测具体答案。
- 问题部分可回答、部分无依据时，区分已知与未知，并明确未知部分无法从当前知识库确认。
- 拒答语气应简洁、友好，可提示用户换一种问法或补充关键词。

### 4.4 数据计数规则
- “文档目录数”指当前知识库中可被系统识别的文档目录/条目数量，具体以实现定义为准，并在界面保持一致。
- 仅上传成功并完成入库的 Markdown 文档计入目录数；上传失败或处理失败的文档不计入。
- 同一文档不得重复计数；刷新页面后计数应保持一致。

## 5. 功能需求

### 5.1 功能清单

| 模块 | 功能点 | 优先级 | 说明 |
|---|---|---:|---|
| 知识库上传 | 选择并上传 Markdown 单文件 | P0 | 仅接受 `.md`/Markdown 文档 |
| 知识库上传 | 格式校验与上传反馈 | P0 | 明确成功、失败和处理中状态 |
| 知识库上传 | Markdown 解析与入库 | P0 | 完成后才进入问答范围 |
| 目录查看 | 查看文档目录/列表 | P0 | 展示已成功入库的文档条目 |
| 目录查看 | 查看文档目录数 | P0 | 展示当前有效文档数量 |
| 知识问答 | 输入问题并提交 | P0 | 支持自然语言问题 |
| 知识问答 | 基于知识库检索并回答 | P0 | 严格受知识库约束 |
| 知识问答 | 引用依据展示 | P0 | 展示文档名称，能力允许时展示片段 |
| 知识问答 | 无依据拒答与模糊问题澄清 | P0 | 禁止无依据猜测 |

### 5.2 功能一：知识库 Markdown 文档上传

#### 5.2.1 目的
将可解析的 Markdown 文档导入知识库，使其成为后续问答的唯一可用知识来源。

#### 5.2.2 规则
- 提供单文件选择和上传入口。
- 仅允许 Markdown 文件；推荐扩展名为 `.md`，实际校验同时依据文件类型和解析结果。
- 上传前校验文件类型；非 Markdown 文件不得进入处理流程，并提示“仅支持 Markdown 文件”。
- 上传成功后进入解析/索引处理；处理完成后才标记为可问答。
- 文档名称应保留，作为目录展示和问答引用的基础标识。
- 空文件、无法解析或内容为空的 Markdown 文档视为处理失败，不进入问答范围。

#### 5.2.3 状态
- 待上传：尚未提交文件。
- 上传中：文件传输进行中，上传入口不可重复提交同一任务。
- 处理中：文件已上传，正在解析和建立检索索引。
- 成功：解析和入库完成，可在目录中看到，并可用于问答。
- 失败：上传、解析或入库失败，展示失败原因；该文档不计入目录数、不参与问答。

#### 5.2.4 边界与异常
- 网络中断、服务异常或超出系统限制时，展示可理解的失败提示。
- 重复上传同名文档的处理策略需在开发前确认；MVP 不提供替换或版本管理。
- 不提供文件夹上传和非 Markdown 格式转换。

### 5.3 功能二：查看知识库文档目录数

#### 5.3.1 目的
让用户快速确认当前知识库已纳入多少有效文档，并可查看对应目录条目。

#### 5.3.2 规则
- 页面展示当前有效文档目录数，数值来源于已成功入库的文档。
- 提供目录/列表视图，至少展示文档名称和可识别的处理状态。
- 处理中和失败文档不得被误显示为可问答文档；失败文档不计数。
- 上传成功或处理完成后，目录数应在刷新或约定的自动更新机制下同步。
- 目录为空时展示 0，并提供上传 Markdown 文档的引导。

#### 5.3.3 状态
- 加载中：目录数和列表尚未返回，展示加载状态。
- 有数据：展示准确目录数和有效文档条目。
- 空态：目录数为 0，提示当前暂无知识库文档。
- 加载失败：展示错误提示和重试入口，不展示过期数值为当前结果。

### 5.4 功能三：知识问答

#### 5.4.1 目的
让用户通过自然语言查询当前知识库，并获得可验证、不过度推断的答案。

#### 5.4.2 规则
- 用户输入问题并提交；空问题不可提交。
- 系统只检索已成功入库的 Markdown 文档。
- 生成答案前必须判断检索内容是否足以支持结论。
- 有充分依据时返回简洁答案，并展示引用文档名称；有片段能力时展示相关片段。
- 只有相关性不足或证据不足时，才返回拒答；不得为了满足用户而补答。
- 问题不清晰时返回澄清提示，指出需要补充的对象、功能、场景或条件。
- 问答结果只针对本次输入，不要求 MVP 维护复杂会话上下文。

#### 5.4.3 问答状态
- 初始态：展示输入框和提问引导。
- 提交中：展示检索/生成中的状态，避免重复提交。
- 有依据回答：展示答案和对应引用依据。
- 无依据拒答：展示知识库无依据的说明和重新提问引导。
- 需澄清：展示需要补充的信息。
- 空知识库：提示当前无可用于问答的文档，并引导先上传 Markdown。
- 系统错误：提示本次问答失败，并提供重试；不输出未经验证的兜底答案。

#### 5.4.4 问答知识库约束
- 知识范围仅限当前有效 Markdown 文档的可检索内容。
- 不允许调用外部搜索、外部知识或模型常识补全。
- 若用户要求总结、比较或推理，结论也必须能由知识库内容直接支持；无法支持时拒答或缩小回答范围。
- 文档之间信息冲突时，不擅自裁决事实；应指出存在冲突并引用相关文档，或转为待确认。
- 涉及知识库未定义的时间、数字、条件或承诺时，不得自行推导为确定结论。

## 6. 用户流程

### 6.1 上传与入库
1. 知识维护者进入知识库页面。
2. 选择一个 Markdown 文件并提交上传。
3. 系统校验格式并展示上传中、处理中状态。
4. 处理成功后将文档加入目录并计入目录数。
5. 处理失败则展示原因，文档不进入目录和问答范围。

### 6.2 查看目录数
1. 用户进入知识库页面。
2. 系统加载目录数和有效文档列表。
3. 用户查看当前文档目录数及文档名称。
4. 无文档时查看空态和上传引导。

### 6.3 知识问答
1. 用户输入问题并提交。
2. 系统检索当前有效 Markdown 文档。
3. 系统判断证据是否足够。
4. 有依据则返回答案和引用；无依据则拒答；不清晰则请求澄清。
5. 用户可基于提示重新提问；MVP 不要求复杂会话管理。

## 7. 验收标准

| 编号 | 验收项 | 标准 |
|---|---|---|
| AC-01 | Markdown 上传 | 可选择并上传 Markdown；非 Markdown 被拦截并提示仅支持 Markdown |
| AC-02 | 上传状态 | 上传中、处理中、成功、失败状态可识别，失败原因可理解 |
| AC-03 | 入库范围 | 仅处理成功的 Markdown 文档进入目录和问答范围 |
| AC-04 | 空文档 | 空文件或无法解析文档处理失败，不计入目录数、不参与问答 |
| AC-05 | 目录展示 | 页面能展示有效文档目录及文档名称 |
| AC-06 | 目录数 | 目录数等于有效入库文档数，不重复计数；空库显示 0 |
| AC-07 | 目录异常 | 加载中、加载失败和空态均有明确展示，失败时可重试 |
| AC-08 | 有依据问答 | 对知识库内可支持的问题返回相关答案，并展示文档名称或可用引用片段 |
| AC-09 | 无依据拒答 | 对知识库外问题明确拒答，不输出外部知识、猜测或伪造依据 |
| AC-10 | 模糊问题 | 对缺少关键信息的问题先澄清，不直接猜测答案 |
| AC-11 | 部分依据 | 仅回答知识库能支持的部分，并明确无法确认的部分 |
| AC-12 | 空库问答 | 无有效文档时不生成知识答案，提示先上传 Markdown |
| AC-13 | 问答异常 | 检索或生成失败时展示错误和重试，不使用未经验证的兜底答案 |
| AC-14 | 约束验证 | 问答流程不读取 PDF、TXT、Word，不调用联网搜索或知识库外内容 |

## 8. 风险与依赖

### 8.1 风险
- Markdown 格式自由度高，标题、表格、代码块等结构可能影响解析和检索质量。
- 文档内容不足或表达冲突会导致无法回答，需坚持拒答规则而非强行补答。
- 目录数口径若与入库状态不一致，会影响用户对知识覆盖范围的判断。
- 检索召回不足会造成“知识库有内容但未命中”，需保留可观测错误信息。
- 模型可能产生超出证据的扩写，需要在生成后进行证据约束或结果校验。

### 8.2 依赖
- Markdown 文件上传、存储、解析和索引服务。
- 文档处理状态及有效文档计数接口。
- 知识检索与答案生成服务。
- 前端对上传、目录、问答各状态的统一展示。
- 测试用的有依据、无依据、模糊、冲突和空知识库样例文档。

## 9. 待确认项

- 单文件大小、文档总量、单次上传频率等系统限制。
- Markdown 文件类型校验规则，以及是否允许无 `.md` 后缀但内容合法的文件。
- 重复上传同名文档时的处理方式：拒绝、忽略或生成重复条目。
- “文档目录数”最终口径：按有效文档数，还是按目录条目/文档节点数统计。
- 目录列表是否展示处理中和失败条目，及其具体文案。
- 引用展示的最小交付形态：仅文档名称，或同时返回命中片段。
- 问答相关性阈值、拒答阈值和冲突文档的展示策略。
- 目录数与问答索引完成后的数据刷新时机。
- 是否需要基础的访问入口保护；本期不设计权限、角色和多租户能力。
