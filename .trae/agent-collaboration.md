# 多代理协作流程说明

本文档定义本项目内的标准协作链路，目标是让 **PM / UED / Tech Lead / Backend / Frontend / QA** 在同一套上下文下稳定协作，减少需求遗漏、接口返工、实现偏差与测试盲区。

---

## 1. 推荐协作链路

默认按以下顺序推进：

1. `pm`：产出产品方案 / PRD
2. `ued`：基于 PRD 产出 UI / 交互方案
3. `tech-lead`：基于 PRD + UI 方案产出技术方案与任务拆解
4. `backend`：依据 tech-lead 的技术方案完成后端实现与单元测试
5. `frontend`：依据 PRD + UED + tech-lead 的技术方案完成前端实现
6. `qa`：基于 PRD / UED / 技术方案 / 实现结果产出测试方案、回归清单与验收建议
7. （可选）主代理：组织联调、验收、回归闭环

适用场景：
- 新功能建设
- 中大型需求改造
- 需要先设计方案、再分角色落地的任务
- 需要降低前后端反复对齐成本的需求

不一定需要完整链路的场景：
- 小型 bug 修复
- 明确且局部的后端/前端单点改动
- 不涉及产品方案和 UI 变化的技术性任务

## 2. 每个代理的职责边界

### 2.1 `pm`
负责：
- 理解业务背景、目标用户、核心场景
- 输出结构化 PRD
- 补齐范围、规则、边界、验收标准

不负责：
- 技术架构设计
- UI 视觉稿输出
- 具体代码实现

核心交付物：
- PRD 文档
- 待澄清问题清单
- 验收标准

### 2.2 `ued`
负责：
- 基于 PRD 输出可实现的 UI / UX 方案
- 明确信息架构、页面结构、交互流程、状态设计
- 输出前端可落地的页面/组件说明

不负责：
- 后端技术设计
- 直接编码实现

核心交付物：
- UED 方案
- 页面/模块设计说明
- 状态与交互规则说明
- 前端交付说明

### 2.3 `tech-lead`
负责：
- 基于 PRD + UED 方案产出完整技术方案
- 设计模块边界、领域模型、流程、接口、数据结构
- 拆解可执行开发任务
- 明确实施优先级、依赖、风险与验收标准

不负责：
- 默认不直接承担大规模编码
- 不替代 PM/UED 的原始职责

核心交付物：
- 技术方案
- 接口设计
- 开发任务拆解
- 研发执行说明

### 2.4 `backend`
负责：
- 基于 tech-lead 已确认的技术方案完成后端实现
- 按既定方案落地数据库 / 领域 / 服务 / API 相关改动
- 编写单元测试
- 输出后端交付说明

不负责：
- 技术方案主设计
- 替代 PM/UED/tech-lead 做上游方案定义
- 在信息不清情况下盲目实现

核心交付物：
- 后端代码
- 单元测试
- 实现说明与验证结果

### 2.5 `frontend`
负责：
- 基于 PRD + UED + tech-lead 技术方案完成前端实现
- 保证交互、状态、组件结构、类型与工程质量
- 输出前端交付说明

不负责：
- 技术方案主设计
- 替代 PM/UED/tech-lead 做上游方案定义
- 自行发明未确认交互

核心交付物：
- 前端代码
- 页面/组件实现
- 前端验证说明

### 2.6 `qa`
负责：
- 基于 PRD、UED、技术方案与实现结果设计测试方案
- 产出测试用例、覆盖矩阵、回归清单与验收建议
- 识别功能、接口、交互、数据、权限、稳定性等质量风险

不负责：
- 默认不承担功能实现
- 不在需求不清时直接给出武断验收结论

核心交付物：
- 测试方案
- 测试用例
- 回归清单
- 风险分析与验收建议

## 3. 标准交付顺序

### 阶段 A：需求定义
由 `pm` 产出：
- 背景与目标
- 用户与场景
- 功能范围
- 业务规则
- 验收标准
- 待确认事项

进入下一阶段的条件：
- 关键业务目标明确
- 本期范围明确
- 核心规则与验收标准基本明确

### 阶段 B：交互与界面设计
由 `ued` 基于 PRD 产出：
- 页面/模块清单
- 主流程与分支流程
- 状态设计（空态/加载态/错误态/成功反馈）
- 页面结构与组件建议
- 前端交付说明

进入下一阶段的条件：
- 核心页面和交互路径明确
- 前端需要的数据与动作明确
- 页面状态与边界条件基本齐备

### 阶段 C：技术方案与任务拆解
由 `tech-lead` 基于 PRD + UED 方案产出：
- 架构设计
- 流程设计
- 模型设计
- 接口设计
- 数据存储设计
- 稳定性/性能/安全设计
- 开发任务拆解
- Backend / Frontend 执行建议

进入下一阶段的条件：
- 核心接口、数据模型、模块边界已定义
- 关键技术风险已识别
- 任务拆解可直接分派实施

### 阶段 D：后端实现
由 `backend` 基于技术方案产出：
- 数据库变更 / 模型实现
- 业务逻辑实现
- API 实现
- 单元测试
- 后端交付说明

### 阶段 E：前端实现
由 `frontend` 基于 PRD + UED + 技术方案产出：
- 页面/组件实现
- 状态管理与接口接入
- 前端验证说明

### 阶段 F：测试与验收
由 `qa` 基于 PRD / UED / 技术方案 / 实现结果产出：
- 测试方案
- 测试用例与覆盖矩阵
- 接口与数据验证点
- 回归测试清单
- 风险分析与验收建议

进入收尾阶段的条件：
- 主流程与关键异常流程已有验证方案
- 高风险问题已识别并分级
- 发布建议与残余风险明确

## 4. 代理之间的输入输出约定

### 4.1 `pm` → `ued`
输入：
- 需求原文 / 背景
- 业务目标
- 用户角色
- 功能范围
- 核心规则

输出建议包含：
- 页面目标
- 用户主路径
- 关键状态
- 异常与边界说明

### 4.2 `ued` → `tech-lead`
输入：
- PRD
- UI / 交互方案
- 页面结构
- 用户动作与状态说明

输出建议包含：
- 页面触发了哪些后端动作
- 页面需要哪些数据对象
- 首屏 / 列表 / 详情 / 提交 / 状态流转等交互诉求

### 4.3 `tech-lead` → `backend`
输入：
- 技术方案
- 接口设计
- 数据模型设计
- 开发任务拆解
- Backend 执行建议

输出建议包含：
- 模块改动范围
- 核心服务职责
- API 落地实现说明
- 单元测试范围

### 4.4 `tech-lead` → `frontend`
输入：
- 技术方案
- 接口设计
- 前端需要的响应结构与状态语义
- Frontend 执行建议

输出建议包含：
- 页面/组件与接口映射关系
- 前端状态处理要求
- 页面实现边界与注意点

### 4.5 `frontend` → `qa`
输入：
- 前端实现说明
- 页面/组件改动范围
- 交互状态与接口接入结果
- 已知限制与验证结果

输出建议包含：
- 页面主流程与异常流程测试用例
- 交互状态覆盖点
- 回归影响范围

### 4.6 `backend` → `qa`
输入：
- 后端实现说明
- API 列表与契约
- 数据模型/数据库变更
- 单元测试范围与结果

输出建议包含：
- 接口测试点
- 数据一致性验证点
- 幂等/异常/权限/回归测试点

### 4.7 `qa` → 主代理 / 研发
输入：
- 测试方案
- 测试用例
- 风险列表
- 验收建议

输出建议包含：
- 阻塞发布问题
- 高风险但可带条件发布问题
- 建议补测项
- 最终验收结论

## 5. 推荐协作原则

### 5.1 先方案、后编码
对于中大型需求，先走：
`pm / ued / tech-lead`
再进入：
`backend / frontend`

### 5.2 先澄清、后落地
若发现以下情况，应先澄清再继续：
- PRD 目标不清
- UI 与规则冲突
- 页面动作缺少后端语义
- 状态流转不完整
- 数据归属不明确
- 验收标准缺失

### 5.3 交付物必须可传递
每个代理输出都应满足“下游角色可直接使用”的要求，避免只输出抽象结论。

### 5.4 接口设计必须贴近交互
接口设计不是简单列 CRUD，而是要能支撑：
- 首屏加载
- 列表筛选/分页/排序
- 详情展示
- 表单提交
- 草稿保存
- 状态流转
- 批量操作
- 上传下载
- 异步任务 / 轮询 / 重试

### 5.5 默认先做 MVP
技术方案与任务拆解默认优先给出：
- MVP 主路径
- 核心能力
- 必要异常处理

增强项与优化项可作为后续阶段追加。

### 5.6 测试要后置但不要缺席
对于完整需求，推荐在 `backend / frontend` 完成后引入 `qa` 做质量审视，至少补齐：
- 主流程测试
- 异常流程测试
- 接口与数据一致性测试
- 回归影响评估
- 验收建议

### 5.7 验收结论必须有依据
`qa` 给出的结论应明确区分：
- 已验证
- 待验证
- 建议验证

若存在阻塞发布问题，应明确说明影响范围、严重程度与建议处理方式。

## 6. 推荐调用方式

### 6.1 纯方案型需求
顺序：
1. 调 `pm`
2. 调 `ued`
3. 调 `tech-lead`

适用于：
- 先出方案，不立即开发

### 6.2 全链路落地型需求
顺序：
1. `pm`
2. `ued`
3. `tech-lead`
4. `backend`
5. `frontend`

适用于：
- 明确要从需求一路走到开发实现

### 6.3 后端主导型需求
顺序：
1. 若已有明确 PRD/UED，可直接 `tech-lead`
2. 再 `backend`
3. 最后按需 `frontend`

适用于：
- 后端接口、领域逻辑、数据模型为主

### 6.4 完整闭环型需求
顺序：
1. `pm`
2. `ued`
3. `tech-lead`
4. `backend`
5. `frontend`
6. `qa`

适用于：
- 明确要从需求一路走到方案、开发、测试、验收的任务

---

## 7. 建议的主代理编排方式

主代理在组织多代理协作时，推荐遵循：

1. 先确认用户要的是：
   - 只要方案
   - 方案 + 拆解
   - 直接开发
   - 开发 + 测试验收
2. 若是中大型需求，优先让 `tech-lead` 先产出方案
3. 只有当方案足够稳定时，再分发给 `backend` / `frontend`
4. 若中途发现方案缺失，回退到 `pm` / `ued` / `tech-lead` 补齐，而不是让实现代理自行拍脑袋补需求
5. 若需求包含验收、回归或发布判断，补充引入 `qa` 产出测试方案与验收建议

---

## 8. 一句话使用建议

- 要产品定义：找 `pm`
- 要交互和页面方案：找 `ued`
- 要技术方案和任务拆解：找 `tech-lead`
- 要后端实现和单测：找 `backend`
- 要前端实现：找 `frontend`
- 要测试方案、回归清单和验收建议：找 `qa`
