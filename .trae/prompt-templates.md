# 多代理提示词模板

本文档提供可直接复用的提示词模板，帮助你快速调用 `tech-lead`、`full-stack` 等子代理开展工作。

---

## 1. 技术方案阶段：调用 `tech-lead`

适用场景：
- 需求刚明确，准备先出技术方案
- 希望先做架构设计、接口设计、任务拆解
- 减少直接开发带来的返工

### 模板 1：基于 PRD + UED 输出技术方案

```text
请你作为 tech-lead，基于以下输入产出完整技术方案：

【PRD / 产品需求】
<粘贴 PM 产出的 PRD>

【UED / UI 与交互方案】
<粘贴 UED 产出的页面结构、交互流程、状态说明>

要求：
1. 先总结你对需求、用户、核心场景、关键交互的理解；
2. 输出完整技术方案，至少包含：
   - 背景与目标
   - 核心交付功能点
   - 架构设计
   - 流程设计
   - 模型设计
   - 详细设计
   - 接口设计
   - 数据存储设计
   - 稳定性 / 性能 / 安全设计
   - 测试与发布建议
   - 开发任务拆解
3. 接口设计要贴合真实前端交互，不要只给 CRUD；
4. 默认后端技术栈优先按 Python + MySQL 考虑；
5. 如果信息不足，先输出待澄清问题，不要直接跳过。
```

### 模板 2：基于现有代码做技术方案落地

```text
请你作为 tech-lead，结合当前项目代码结构、以下需求和 UI 设计，产出一个可直接指导研发落地的技术方案。

【需求】
<粘贴需求>

【UI / 交互】
<粘贴 UI 方案>

要求：
1. 先检查现有项目技术栈、目录结构、分层模式和可复用能力；
2. 输出贴合当前仓库的技术方案，而不是泛化架构；
3. 必须包含接口设计、数据模型设计、任务拆解；
4. 明确说明 full-stack 需要完成的后端实现、前端实现、单元测试与 E2E 测试范围；
5. 输出 MVP 优先级和后续增强建议。
```

## 2. 全栈实施与测试阶段：调用 `full-stack`

适用场景：
- 技术方案已基本确定
- 需要统一完成后端 API、业务逻辑、数据库变更、前端页面/组件、接口接入、单元测试与 E2E 测试

### 模板 1：按技术方案完成全栈实现

```text
请你作为 full-stack，基于以下技术方案完成前后端实现、单元测试与 E2E 测试。

【技术方案】
<粘贴 tech-lead 产出的技术方案>

重点要求：
1. 先总结你理解到的业务目标、关键流程、接口范围、页面范围、数据模型与测试目标；
2. 检查当前项目中可复用的后端模块、前端组件、分层、基础设施与测试设施；
3. 按方案完成后端代码实现、前端页面/组件实现与接口接入；
4. 编写单元测试，覆盖核心业务规则、关键异常分支、边界条件与前端关键交互逻辑；
5. 编写 E2E 测试，覆盖用户主流程、关键异常流程、空态/加载态/错误态和回归链路；
6. 输出全栈交付说明、验证结果、已知限制与验收建议；
7. 明确区分：已验证 / 未执行 / 建议验证；
8. 如发现方案与现有代码冲突，先指出冲突和建议，不要直接硬改。
```

### 模板 2：按 PRD + UED + 技术方案完成全栈落地

```text
请你作为 full-stack，深度理解下面的 PRD、UED 交互说明和 tech-lead 技术方案，基于既定方案完成前后端代码实现、单元测试与 E2E 测试。

【PRD】
<粘贴 PRD>

【UED】
<粘贴 UED 方案>

【技术方案】
<粘贴技术方案>

要求：
1. 以 tech-lead 已确认的接口设计、模型设计和任务拆解为直接依据，不自行扩展成新的总体技术方案；
2. 默认后端技术栈优先采用 Python + MySQL，前端 Web 应用优先采用 TypeScript；若项目已有技术栈则遵循项目上下文；
3. 后端实现需补齐请求参数校验、响应结构落地、错误码处理、分页/过滤/排序、幂等性、性能与可靠性实现细节；
4. 前端实现需补齐页面/组件结构、状态管理、接口接入、表单与交互反馈、错误处理、可访问性与响应式适配；
5. 测试需覆盖单元测试与 E2E 测试，不能只给测试建议替代应实现的测试；
6. 如发现方案与现有代码冲突，先指出冲突和建议，不要直接硬改。
```

### 模板 3：针对当前实现做质量评审与补测

```text
请你作为 full-stack，对当前实现做一次质量评审，并补齐缺失的单元测试与 E2E 测试。

输入如下：
【PRD】
<粘贴 PRD>

【UED】
<粘贴 UED>

【技术方案】
<粘贴技术方案>

【当前实现说明】
<粘贴当前实现说明、接口列表、改动范围、已知限制、验证结果>

要求：
1. 对照 PRD、UED 和技术方案，检查实现是否存在遗漏、偏差和风险；
2. 对发现的问题直接修复，除非需要回到 PM/UED/tech-lead 确认；
3. 补齐必要的单元测试与 E2E 测试；
4. 输出测试覆盖矩阵、缺陷清单、回归清单和验收建议；
5. 对每个高风险问题补充影响范围、触发条件、复现路径和严重程度；
6. 明确哪些问题会阻塞发布，哪些问题可以带条件发布。
```

## 3. 全链路协作模板

适用场景：
- 你想一次性明确整个推进顺序
- 希望主代理按固定链路组织多个 subagent

### 模板：先方案后开发再验收

```text
请按以下顺序协作推进这个需求：
1. 先让 pm 产出 PRD；
2. 再让 ued 基于 PRD 产出 UI / 交互方案；
3. 再让 tech-lead 基于 PRD + UED 方案产出技术方案、接口设计与任务拆解；
4. 然后让 full-stack 按技术方案完成前后端实现、单元测试与 E2E 测试，并输出回归清单与验收建议。

需求如下：
<粘贴需求>

要求：
- 每个阶段先给出结构化交付物，再进入下一阶段；
- 若上游信息不足，下游不得自行臆测，需明确提出待澄清项；
- 默认后端技术栈优先采用 Python + MySQL；
- 在 full-stack 交付阶段明确区分已验证、未执行、建议验证。
```

## 4. 极简调用模板

如果你只是想快速触发某个 agent，可以直接用以下短模板。

### 调 `tech-lead`

```text
请作为 tech-lead，基于以下需求产出完整技术方案、接口设计和开发任务拆解：
<需求内容>
```

### 调 `full-stack`

```text
请作为 full-stack，基于以下 PRD、UED 和 tech-lead 技术方案完成前后端实现、单元测试与 E2E 测试：
<输入内容>
```

---

## 5. 使用建议

- 中大型需求：优先 `tech-lead` 先出方案，再进 `full-stack` 做前后端实现、单元测试与 E2E 测试
- 后端主导需求：可直接 `tech-lead` → `full-stack`，仍由 `full-stack` 维护接口契约和必要前端联动
- 已有成熟方案：可跳过上游，直接把确认过的 PRD / UED / tech-lead 方案交给 `full-stack`
- 一旦发现 PRD、UED、技术方案三者冲突，优先回到上游补齐，不建议让实现代理自行猜测
- 若目标包含上线评估、回归确认、发布判断，要求 `full-stack` 在交付说明中明确测试覆盖、阻塞问题、残余风险与验收建议
