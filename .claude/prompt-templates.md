# 多代理提示词模板

本文档提供可直接复用的提示词模板，帮助你快速调用 `tech-lead`、`backend`、`frontend` 等子代理开展工作。

---

## 1. 技术方案阶段：调用 `tech-lead`

适用场景：
- 需求刚明确，准备先出技术方案
- 希望先做架构设计、接口设计、任务拆解
- 减少直接开发带来的返工

### 模板 1：基于 PRD + UED 输出技术方案

```text
请你作为 tech-lead，基于以下输入产出完整技术方案：

【PRD / 产品需求】
<粘贴 PM 产出的 PRD>

【UED / UI 与交互方案】
<粘贴 UED 产出的页面结构、交互流程、状态说明>

要求：
1. 先总结你对需求、用户、核心场景、关键交互的理解；
2. 输出完整技术方案，至少包含：
   - 背景与目标
   - 核心交付功能点
   - 架构设计
   - 流程设计
   - 模型设计
   - 详细设计
   - 接口设计
   - 数据存储设计
   - 稳定性 / 性能 / 安全设计
   - 测试与发布建议
   - 开发任务拆解
3. 接口设计要贴合真实前端交互，不要只给 CRUD；
4. 默认后端技术栈优先按 Python + MySQL 考虑；
5. 如果信息不足，先输出待澄清问题，不要直接跳过。
```

### 模板 2：基于现有代码做技术方案落地

```text
请你作为 tech-lead，结合当前项目代码结构、以下需求和 UI 设计，产出一个可直接指导研发落地的技术方案。

【需求】
<粘贴需求>

【UI / 交互】
<粘贴 UI 方案>

要求：
1. 先检查现有项目技术栈、目录结构、分层模式和可复用能力；
2. 输出贴合当前仓库的技术方案，而不是泛化架构；
3. 必须包含接口设计、数据模型设计、任务拆解；
4. 明确说明哪些由 backend 执行，哪些由 frontend 执行；
5. 输出 MVP 优先级和后续增强建议。
```

## 2. 后端实施阶段：调用 `backend`

适用场景：
- 技术方案已基本确定
- 需要后端实现 API、业务逻辑、数据库变更、单元测试

### 模板 1：按技术方案实现后端

```text
请你作为 backend，基于以下技术方案完成后端实现。

【技术方案】
<粘贴 tech-lead 产出的技术方案>

重点要求：
1. 先总结你理解到的业务目标、关键流程、接口范围、数据模型；
2. 检查当前项目中可复用的模块、分层、基础设施与测试设施；
3. 按方案完成后端代码实现；
4. 编写单元测试，覆盖核心业务规则、关键异常分支与边界条件；
5. 输出改动说明、验证结果、已知限制；
6. 如发现方案与现有代码冲突，先指出冲突和建议，不要直接硬改。
```

### 模板 2：重点做接口设计 + 编码

```text
请你作为 backend，深度理解下面的 PRD、UED 交互说明和技术方案，重点完成：接口设计细化、后端代码实现、单元测试。

【PRD】
<粘贴 PRD>

【UED】
<粘贴 UED 方案>

【技术方案】
<粘贴技术方案>

要求：
1. 接口设计必须围绕真实前端交互场景；
2. 默认技术栈优先采用 Python + MySQL；
3. 必须补齐请求参数、响应结构、错误码、分页/过滤/排序、幂等性、性能与可靠性设计；
4. 完成实现后提供后端交付说明。
```

## 3. 前端实施阶段：调用 `frontend`

适用场景：
- PRD、UED、接口设计已明确
- 需要完成页面/组件/状态管理与接口接入

### 模板 1：按 UED + 接口设计实现前端

```text
请你作为 frontend，基于以下输入完成前端实现：

【PRD】
<粘贴 PRD>

【UED 方案】
<粘贴 UED 方案>

【接口设计 / 技术方案】
<粘贴 tech-lead 的接口设计或技术方案>

要求：
1. 先总结你理解到的页面结构、核心交互、状态要求；
2. 检查当前项目中的组件、hooks、状态管理和样式方案，优先复用；
3. 完成页面/组件/交互实现；
4. 补齐空态、加载态、错误态、成功反馈、权限态、边界态；
5. 输出前端交付说明与验证结果。
```

### 模板 2：只做前端工程落地

```text
请你作为 frontend，基于已确认的产品与设计方案，在当前项目中落地实现该功能。

你需要重点关注：
- 组件拆分
- 数据流与状态管理
- 接口接入
- 表单与交互反馈
- 错误处理
- 可访问性与响应式适配

如果发现接口设计与页面交互不匹配，请先结构化指出问题和建议。
```

## 4. 全链路协作模板

适用场景：
- 你想一次性明确整个推进顺序
- 希望主代理按固定链路组织多个 subagent

### 模板：先方案后开发

```text
请按以下顺序协作推进这个需求：
1. 先让 pm 产出 PRD；
2. 再让 ued 基于 PRD 产出 UI / 交互方案；
3. 再让 tech-lead 基于 PRD + UED 方案产出技术方案、接口设计与任务拆解；
4. 然后让 backend 按技术方案完成后端实现与单元测试；
5. 最后让 frontend 完成前端实现。

需求如下：
<粘贴需求>

要求：
- 每个阶段先给出结构化交付物，再进入下一阶段；
- 若上游信息不足，下游不得自行臆测，需明确提出待澄清项；
- 默认后端技术栈优先采用 Python + MySQL。
```

## 5. 极简调用模板

如果你只是想快速触发某个 agent，可以直接用以下短模板。

### 调 `tech-lead`

```text
请作为 tech-lead，基于以下需求产出完整技术方案、接口设计和开发任务拆解：
<需求内容>
```

### 调 `backend`

```text
请作为 backend，基于以下技术方案完成后端实现与单元测试：
<技术方案内容>
```

### 调 `frontend`

```text
请作为 frontend，基于以下 PRD、UED 和接口设计完成前端实现：
<输入内容>
```

---

## 6. 使用建议

- 中大型需求：优先 `tech-lead` 先出方案，再进 `backend` / `frontend`
- 后端主导需求：可直接 `tech-lead` → `backend`
- 已有成熟方案：可跳过上游，直接把确认过的内容交给 `backend` / `frontend`
- 一旦发现 PRD、UED、技术方案三者冲突，优先回到上游补齐，不建议让实现代理自行猜测
